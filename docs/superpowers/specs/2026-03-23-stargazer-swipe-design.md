# StarGazer Swipe — Design Spec

A Tinder-like swiping interface for discovering GitHub repos starred by your network. Deployed as a static GitHub Pages site, auto-generated by the existing newsletter pipeline.

## Platform

**GitHub Actions + GitHub Pages (auto-deploy).** The existing `newsletter.py` cron pipeline is extended to also generate a self-contained `index.html`. A new workflow step deploys it to GitHub Pages via the `gh-pages` branch.

## Architecture

```
newsletter.py
  ├── fetch network stars, metadata, topics (existing)
  ├── fetch README markdown per repo (new)
  ├── generate insights per repo (new, see Insights Pipeline)
  ├── create GitHub issue (existing)
  └── generate dist/index.html with all data embedded as JSON (new)

GitHub Actions workflow
  ├── run newsletter.py (existing step)
  └── deploy dist/ to GitHub Pages (new step)
```

Single `index.html` output. All repo data (metadata, README HTML, insights) is serialized as a JSON blob inside a `<script>` tag. No external data fetches at runtime.

## Insights Pipeline

Three-tier fallback to minimize token cost:

```
For each repo:
  1. DeepWiki (free, no tokens)
     GET https://api.deepwiki.com/v1/repos/{owner}/{repo}/summary
     ├── If 200 with content → use as insight
     ├── If 404 or empty → try HTML scrape of https://deepwiki.com/{owner}/{repo}
     └── If neither works → fallback to step 2
  2. LLM (if API key set, costs tokens)
     ├── Send repo README (truncated to 4000 chars) + metadata as context
     └── Generate structured insight
  3. Heuristics only (no API key, no DeepWiki)
     └── No insight box shown on card
```

**DeepWiki integration note:** DeepWiki may not have a public API. Implementation should first check for a JSON API, then fall back to HTML scraping of the page intro section. If neither is reliable, skip DeepWiki and go straight to LLM. This is a known risk — a feasibility check during implementation will determine the extraction method.

### AI provider selection

Priority order based on which secret is set:

1. `ANTHROPIC_API_KEY` → Claude
2. `OPENAI_API_KEY` → OpenAI
3. `GEMINI_API_KEY` → Gemini
4. None → heuristics only

### Insight content

Each insight covers three things:

- **What it does** — core purpose in one sentence
- **Spotlight** — what makes it stand out (unique approach, key feature, differentiator)
- **Summary** — who it's for, when you'd use it

DeepWiki intros naturally cover this. The LLM fallback prompt requests the same structure.

### Token optimization

- DeepWiki is tried first for every repo (free)
- LLM calls are batched in chunks of 10 repos max per request (avoids context window limits)
- Each repo's README is truncated to 4000 chars before sending to LLM
- Expected: most popular repos are on DeepWiki, LLM only runs for niche/new repos

## Card Design

**Rich card with AI insight box.** Dark theme, Inter font.

### Card fields

| Field | Source |
|-------|--------|
| Repo name & owner | GitHub API (existing) |
| Description | GitHub API (existing) |
| Topics (max 3) | GitHub API (existing) |
| Star count | GitHub API (existing) |
| Network users who starred | Star events (existing) |
| AI Insight box | DeepWiki / LLM / hidden |

The AI Insight box is visually distinct (gradient background, purple accent). It only renders when insight data exists.

## Layout

### Desktop (>768px)

**40/60 horizontal split:**

- **Left panel (40%):** Info card with repo metadata, AI insight, swipe buttons
- **Right panel (60%):** Pre-rendered README (markdown converted to HTML at build time via GitHub API `Accept: application/vnd.github.html+json`)

### Mobile (<=768px)

**Full-width info card only.** No README panel. Touch swipe gestures supported.

## Interactions

| Action | Desktop | Mobile |
|--------|---------|--------|
| Skip (next card) | Click X button / press left arrow | Swipe left / tap X |
| Like (open repo) | Click checkmark / press right arrow | Swipe right / tap checkmark |

**Swipe right behavior:** Opens `https://github.com/{owner}/{repo}` in a new browser tab. User can star it manually on GitHub.

**Progress indicator:** Top bar showing "Card N of Total".

## End Screen

**Summary page** displayed after all cards are swiped:

- List of all liked repos with direct GitHub links
- Stats: total repos seen, liked count, skipped count

## Data Flow (build time)

1. Pipeline collects repos from network star events (existing `get_user_star_events`)
2. For each repo, fetch metadata via `get_repo_metadata` (existing, cached)
3. For each repo, fetch README HTML via GitHub API
4. For each repo, attempt DeepWiki intro fetch
5. For repos without DeepWiki data, batch LLM call if API key is set
6. Serialize all data as JSON, inject into HTML template
7. Write `dist/index.html`

## Workflow Changes

```yaml
# Add to newsletter.yml after the existing "POST newsletter" step:

- name: Deploy to GitHub Pages
  uses: peaceiris/actions-gh-pages@v4
  with:
    github_token: ${{ secrets.GITHUB_TOKEN }}
    publish_dir: ./dist
```

New env vars (all optional):

```yaml
env:
  ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
  GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
```

Workflow permissions need `contents: write` (for pushing to `gh-pages` branch).

## Accessibility

- Keyboard navigation: arrow keys for swiping, Enter to confirm
- Focus states on all interactive elements
- `prefers-reduced-motion` respected (disable swipe animations)
- Minimum 4.5:1 contrast ratio for all text
- SVG icons only, no emoji as UI icons

## Error Handling & Graceful Degradation

- **README fetch fails:** Card renders without README panel (info card only, like mobile)
- **DeepWiki unavailable:** Skip silently, fall through to LLM or heuristics
- **LLM returns malformed JSON:** Log warning, skip insight for those repos
- **GitHub API rate limited:** Use cached metadata where available; README fetch uses `If-None-Match` headers for conditional requests
- **README too large:** Truncate to 50,000 chars per repo to keep `index.html` reasonable

## Implementation Notes

### HTML template

The HTML template is a Python string in `newsletter.py` (no external template engine). A `TEMPLATE` constant holds the full HTML/CSS/JS with a `{{DATA}}` placeholder replaced by `json.dumps()`.

### Touch swipe

Vanilla JS `touchstart`/`touchmove`/`touchend` listeners. Threshold of 80px horizontal movement to trigger swipe. CSS `transform: translateX()` for swipe animation. Vertical scrolling is not blocked.

### README styling

Include `github-markdown-css` (CDN link) scoped to the README panel via a `.markdown-body` wrapper class. This gives GitHub-like rendering without custom CSS.

### JSON data schema

```json
{
  "repos": [
    {
      "name": "owner/repo",
      "description": "...",
      "topics": ["a", "b", "c"],
      "stars": 12345,
      "network_users": ["alice", "bob"],
      "readme_html": "<div>...</div>",
      "insight": "..." | null
    }
  ]
}
```

## Out of Scope

- User authentication in the browser
- Automated starring via API (user stars manually on GitHub)
- Persistent user preferences across sessions
- Multiple pages or routing
